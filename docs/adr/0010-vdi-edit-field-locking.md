# A VDI's submission-defining fields lock once it leaves NOT_STARTED

Editing a Vendor Data Item through the UI (and any direct `PATCH /api/vdi/{id}` caller) is split into two tiers. The descriptive fields — `name`, `submittal_number`, `description`, `spec_drawing_reference` — are editable at any point in the VDI's lifecycle. The submission-defining fields — `item_number`, `approval_type`, `submit_code` — are editable only while the VDI's status is `NOT_STARTED`. Once the VDI has been submitted (status is anything other than `NOT_STARTED`), an attempt to change a locked field is rejected with `409 Conflict`. The rejection fires only on an *actual value change*: a no-op resubmit of the same value passes, so read-modify-write API clients are not punished. The lock lives in the route layer, not only the UI — the edit modal simply omits the locked fields once the VDI is submitted, but the server is the real guard.

We chose this because these three fields define the *terms of a submittal that has already gone to the buyer*. The buyer received the item under a specific item number, approval type, and submit code; silently rewriting them after the fact would make the stored record disagree with what was actually submitted, and Revisions (per [ADR 0002](0002-revisions-are-lifecycle-driven.md)) preserve that history. Descriptive fields carry no such commitment — fixing a typo in a name or adding a spec reference never contradicts the submittal that went out. The trade-off is giving up the uniform "one always-on Edit button, every field editable" shape that Project uses; we accept the asymmetry because a Project has no lifecycle to protect and a submitted VDI does. This decision extends ADR 0002's principle that a VDI's submitted state is guarded, not freely mutable.

## Considered options

- **Single always-on edit (Project parity)** — one Edit button, all fields editable at any status. Rejected: lets a submitted VDI's item number / approval type / submit code be rewritten after the buyer has already acted on them, so the record drifts from what was submitted.
- **UI-only locking** — hide the fields in the modal but leave the `PATCH` endpoint permissive. Rejected: MCP and PAT-authenticated machine clients (ADR 0008) bypass the UI, so the integrity guarantee would be cosmetic.
- **Two-tier lock enforced in the route layer (chosen)** — descriptive fields always editable; submission-defining fields locked after `NOT_STARTED`, enforced server-side, rejecting only actual changes.
